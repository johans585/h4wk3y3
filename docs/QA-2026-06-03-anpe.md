# QA end-to-end — scan `anpe.bj` — 2026-06-03

Test complet A→Z du pipeline + persistance + dashboard sur le constituent
`anpe.bj`, DB repartie de zéro. Verdict : **le pipeline est fragile face à une
dégradation DNS et plusieurs bugs de persistance/UI font diverger ce que voit
l'analyste de ce qui a été réellement trouvé.**

> Contexte environnement : sur cette VM Kali, le DNS sortant (UDP/TCP 53 vers
> 8.8.8.8 et résolveurs publics) est **refusé** (`connection refused`). C'est le
> déclencheur du scénario ci-dessous. Mais l'incident révèle des défauts de
> robustesse réels, indépendants de l'environnement.

---

## Résultat brut du scan

```
m01 OSINT          ✅ 74s   +4 findings
m02 Subdomain      ❌ TIMEOUT 900s — ABORTÉ (34 subs passifs trouvés puis perdus)
m03 HTTP validator ✅ 0.3s  → 0 live host (entrée vide)

```

11 modules sur 14 n'ont jamais tourné. Le scan se termine en `exit 0` (succès
apparent) alors qu'il n'a produit aucune reconnaissance exploitable.

## Findings (par criticité)

### 🔴 CRITIQUE — Un timeout dans m02 fait perdre TOUT le scan

- m02 trouve 34 sous-domaines passifs en ~46 s, puis bloque ~14 min dans la
  phase de résolution DNS A et atteint le budget module de 900 s.
- Sur `asyncio.TimeoutError`, le pipeline (`core/pipeline.py:447`) abandonne le
  module **sans persister les résultats partiels**. Les 34 subs sont jetés,
  `subdomains.txt` n'est jamais écrit, m03 reçoit une entrée vide → cascade de
  SKIP sur m04→m14.
- **Cause du hang** : `modules/m02_subdomain.py::_run_dnsx_a` fait, après la
  passe bulk (déjà 2 rounds dans `resolve_subs_parallel`), une **3e passe de
  retry inconditionnelle sur TOUS les subs non résolus** à `concurrency=3,
  timeout=10, retries=2` (commentaire : *"exhaustivity beats speed"*). Avec des
  résolveurs lents/injoignables, chaque sub coûte jusqu'à 30 s × (N/3) → la
  phase DNS peut à elle seule dépasser 900 s. Aucune deadline globale ne borne
  cette phase.
- **Impact prod** : il suffit d'un résolveur mort, d'un rate-limit UDP ou d'un
  blip réseau pour rejouer ce scénario sur une vraie cible et perdre l'intégralité
  du scan.

**Correctifs proposés :**
1. Borner la phase DNS par une deadline propre (`asyncio.wait_for` sur le bloc
   résolution) bien inférieure au budget module, et continuer avec ce qui a
   résolu.
2. Persister les résultats partiels avant les sous-étapes risquées (passer les
   subs passifs à m03 même si l'enrichissement DNS échoue).
3. Rendre la 3e passe conditionnelle (ratio de drop > seuil) au lieu
   d'inconditionnelle, et plafonner le nombre de subs re-tentés.
4. Côté pipeline : sur timeout module, récupérer/persister l'état partiel du
   module plutôt que de tout jeter.

### 🟠 HAUTE — Faux positifs SPF/DMARC quand le DNS échoue

`findings.json` contient simultanément, pour le même domaine :
- `email_spoofable` HIGH « SPF record missing » **et**
- `misconfiguration` INFO « SPF check unverified — DNS query failed for anpe.bj ».

Les deux se contredisent : si la requête DNS a échoué, on ne peut pas conclure
que l'enregistrement est *absent*. L'analyse email security classe un échec de
résolution comme « enregistrement manquant → spoofable HIGH ». En prod, sur un
hoquet DNS transitoire, ça génère de faux HIGH.

**Correctif :** un échec de résolution doit dégrader en `unverified` (info), pas
émettre un HIGH. Ne produire « missing → spoofable » que sur un `NOERROR` avec
réponse vide confirmée.

### 🟠 HAUTE — Findings perdus à la persistance DB (6 → 3)

`scan_summary.json` annonce **6 findings (dont 2 HIGH)**, la DB n'en contient que
**3, 0 HIGH**. Le dashboard, qui lit la DB (« source de vérité »), n'affiche
jamais les 2 HIGH.

- **Cause** : `email_spoofable` est dans `ATOMIC_FINDING_TYPES`
  (`core/database.py:51`). Le fingerprint de dedup devient
  `anpe.bj|email_spoofable||atomic` — **identique pour SPF, DMARC et DKIM**. Les
  3 findings s'écrasent (last-write-wins), et c'est le MEDIUM DKIM qui survit, les
  2 HIGH étant écrasés.
- SPF / DMARC / DKIM sont 3 problèmes distincts qui coexistent légitimement sur
  un domaine : les traiter comme « atomiques par (domaine, type, url) » est
  incorrect.

**Correctif :** retirer `email_spoofable` de `ATOMIC_FINDING_TYPES`, ou inclure
un discriminant (selecteur/mécanisme : spf|dmarc|dkim) dans le fingerprint.
(À noter : un 3e finding `misconfiguration` n'a pas non plus persisté — à
investiguer séparément, hors collapse atomique.)

### 🟡 MOYENNE — Overview dashboard : split-brain de sources de données

La page Overview mélange deux sources :
- LIVE HOSTS (20), URLS (1824), TECH STACK → lus depuis les JSON sur disque
  (`output/<domaine>/`).
- SUBDOMAINS, FINDINGS, severity distribution → lus depuis la DB.

Après reset DB, l'écran affiche des chiffres **contradictoires** : `SUBDOMAINS 0`
mais `LIVE HOSTS 20`, `FINDINGS 0` mais un tech stack peuplé (Apache 19…), pour
`adpme.bj` (scan du 21 mai). Un analyste ne peut pas faire confiance à ce tableau.

**Correctif :** une seule source de vérité par page (la DB), ou bannière
explicite « données disque, hors DB ».

### 🟡 MOYENNE — `h4wk3y3.py -t <domaine>` ne crée pas de ligne `targets`

Après le scan d'`anpe.bj` : `scans=1`, `findings=3`, mais `targets=0`. Donc
`/api/targets`, `/api/orgs`, `/api/attack-surface/hosts` renvoient `[]` et les
findings sont orphelins de l'inventaire. Le hook d'attribution post-scan n'a rien
à quoi rattacher. Le quick-start README (`python h4wk3y3.py -t example.com --full`
puis dashboard) donne donc une UI quasi-vide.

**Correctif :** upsert d'une ligne `targets` (apex) au début du scan si absente.

### 🔵 BASSE / hygiène

- **`alembic_version = 0001` alors que le schéma réel est en `0005`** (13 tables,
  toutes les colonnes CVE présentes). Le schéma a été créé par `create_all`, le
  stamp alembic n'a pas suivi → `alembic upgrade head` rejouera 0002→0005 sur un
  schéma déjà à jour et cassera. Stamp à faire : `alembic stamp head`.
- **`testssl.sh` absent du PATH** → m08 SKIP systématique (TLS audit jamais
  exécuté). À installer ou documenter comme dépendance optionnelle.
- **`--help` dit « All 8 modules »** alors qu'il y en a 14+. Texte d'aide obsolète.
- **Login page** : 1 erreur console (`GET /api/auth/me 401`) avant login — bénin
  mais à avaler silencieusement.
- **Warnings de couverture** : `PDCP_API_KEY` absent (chaos off), subfinder
  0/39 providers avec clé → ~30 % de couverture passive. Opérationnel, à corriger
  via clés API.

## Ce qui marche bien

- Auth : 401 enforced sur toute l'API sans session, login form → cookie signé +
  CSRF token, rôles OK (`super-admin` retourné).
- Bootstrap super-admin recréé proprement au 1er boot (`data/.first_admin`).
- m01 OSINT robuste (skips propres quand clés absentes).
- Pipeline ne crash pas : les SKIP en cascade sont gérés sans exception.
- Front soigné (thème ops-center sombre, cohérent), SPA réactive, navigation
  fluide, endpoints API répondent en 200 avec session.
- Idempotence/dedup par fingerprint fonctionne (le problème est qu'elle est trop
  agressive sur les types atomiques, pas qu'elle est absente).

## Limite de ce test

Le DNS sortant bloqué sur cette VM a empêché un scan riche (m03→m14 jamais
exercés sur de vraies données). Les bugs de persistance (#3) et d'UI (#4) sont
néanmoins indépendants de l'environnement et reproductibles. Pour exercer le
pipeline complet : débloquer le DNS sortant ou pointer `subdomain.dns_nameservers`
vers un résolveur joignable.

## ✅ Corrections appliquées (2026-06-03)

Tous les findings ci-dessus ont été corrigés et vérifiés.

**#1 — m02 timeout / perte totale** (`modules/m02_subdomain.py`, `config/h4wk3y3.yaml`)
- Tout le travail DNS post-passif (MX/TXT + email + A/CNAME/PTR) est désormais
  enveloppé dans **un seul budget borné** (`dns_enrich_budget_sec`, défaut 300s).
  Sur dépassement → fallback sur la liste passive non résolue, le **save est
  toujours atteint**.

  bulk ne résout rien (resolver injoignable).

  **34 sous-domaines sauvés** (avant : 0, `subdomains.txt` absent).

  `metadata.record`), folddé dans la clé. Format inchangé quand absent
  (rétro-compatible, aucune régression de dedup cross-scan).
- *Preuve* : sur DB live, `spf_unverified` et `dmarc_unverified` (evidence
  identique, hash `68a2…`) persistent en **2 lignes distinctes**. + 2 tests de

**#3 — FP SPF/DMARC sur échec DNS** (`modules/m01_osint.py`)

## 🌍 Cause racine de l'environnement : Mullvad Lockdown Mode

Le blocage DNS qui déclenchait tout (timeout m02, FP SPF/DMARC) n'était PAS un
bug h4wk3y3 mais l'environnement : **Mullvad VPN connecté avec *Lockdown Mode***
(killswitch). Lockdown bloque tout trafic hors tunnel → les résolveurs publics
qu'h4wk3y3 codait en dur (`8.8.8.8`/`1.1.1.1`/`9.9.9.9`) renvoient
`connection refused`. Seul le résolveur Mullvad `100.64.0.7` (= résolveur
système, `/etc/resolv.conf`) fonctionne.

**Fix robustesse** (`core/dns_resolver.py`, `modules/m01_osint.py`)
- `default_nameservers()` : préflight une fois par process. Résolveurs publics
  s'ils répondent (préférés, anti-split-horizon), **sinon bascule sur le
  résolveur système** (reachable sous VPN killswitch / egress filtré / captive
  portal). Les 6 helpers + m01 l'utilisent au lieu de la liste publique en dur.
- *Preuve* : `default_nameservers()` → `['100.64.0.7']` ; `resolve_a('anpe.bj')`
  → `137.255.10.166` ; SPF/DMARC résolus.

**Résultat — pipeline complet enfin exercé** (scan `--fast`, DNS réparé) :
- m02 : **98s** (avant : timeout 900s) · m03 : **20 live hosts** (avant : 0)
- DB anpe.bj : **subdomains 34 · live_hosts 20 · findings 249** (5 high / 5 med)
- m01 : SPF/DMARC « missing HIGH » **disparus** (FP confirmés — anpe.bj publie
  `v=spf1 … -all`). Ne reste que DKIM medium (légitime) + WHOIS.

*Note opérateur* : si tu préfères forcer un résolveur précis sous VPN, mets
`subdomain.dns_nameservers: ["100.64.0.7"]` (et `osint.dns_nameservers`) dans
`h4wk3y3.yaml` — mais le préflight le fait désormais automatiquement.

## 🖥️ Round 2 — Affichage front « rien ne s'affiche / pas d'organisations »

**Cause #1 (principale) : la table `organisations` était vide.** Le script
`import_data.py` (lit `data/constituents.csv`) n'avait pas été lancé. Exécuté →
**86 orgs / 197 targets** chargés. Le front affiche désormais les constituents.

**Vérif exhaustive des 47 routes API** : toutes **200**, données exactes
(`/api/orgs`=86, `/api/findings/stats` = DB, `/api/orgs/ANPE/stats` live_hosts=20
findings=249, `/api/summary/anpe.bj` cohérent, `/api/subdomains` renvoie
`{count:34,…}`, etc.).

**Bugs de données trouvés & corrigés :**

1. **Attribution NULL sur live_hosts/subdomains après scan interrompu**
   (`core/database.py`). Les findings avaient un fallback d'attribution au save,
   pas les hosts/subs → `attributed_apex` NULL → `/api/attack-surface/hosts?org=`
   vide et `orgs/{}/stats` `live_hosts:0`. Fix : même fallback (= apex du scan) à
   l'insertion dans `upsert_subdomains` / `upsert_live_hosts` (+ `COALESCE` à
   l'update pour ne pas écraser la résolution post-scan). Données existantes
   réparées via `scripts/backfill_attribution.py`.

2. **`/api/domains` listait les dossiers disque** → le front atterrissait sur un
   domaine périmé (`adpme.bj`, scan du 21 mai, affichage incohérent). Fix
   (`dashboard/backend/app.py`) : DB-first — domaines avec scan DB d'abord
   (`has_data:true`), dossiers `output/` orphelins après (`stale:true`). Le front
   ouvre maintenant sur `anpe.bj` avec données cohérentes (34/20/249).

3. **Isolation des tests : `test_org_cli.py` polluait `argus_main`** (subprocess
   `h4wk3y3 org add` → lisait `h4wk3y3.yaml` → DB prod, créait des orgs `cli-*` à
   chaque run). Fix : `_run` propage `ARGUS_TEST_POSTGRES_URL` → `ARGUS_DB_URL`
   (déjà honoré par `resolve_db_url`) → les tests visent la DB isolée. Vérifié :
   suite complète → `argus_main` reste à 86/197.

**Nettoyage** : suppression du junk de test accumulé (`cli-*` orgs, `*.example`
targets). État final propre : **86 orgs / 197 targets** = exactement le CSV.

**Validation** : 739 tests passent ; front vérifié (Playwright) — page
Organisations « 86 organisations · 197 targets », Overview anpe.bj cohérent,
0 erreur console.

## 🚀 Round 3 — 7 axes d'amélioration implémentés

**#1 Résilience cycle de vie scan** (`core/pipeline.py`, `core/database.py`)
- Exécution modules + attribution + `finish_scan` enveloppés en `try/except/finally`
  → un scan interrompu (exception, Ctrl-C, cancel asyncio) est TOUJOURS finalisé,
  marqué `status='partial'`, attribué et persisté. Plus de scan fantôme `running`.
- `finish_scan(status=)` + `abandon_stale_scans(older_than_hours=6)` (age-gated, sûr
  vis-à-vis d'un scan concurrent) appelé au démarrage dashboard + CLI.

**#2 Source unique DB** (`dashboard/backend/app.py`)
- `/api/subdomains` (statut par-host) et `/api/tech` lisent désormais la DB (fallback
  disque gaté sur « aucun scan DB »). Le split-brain résiduel est fermé. Les endpoints
  restants (urls/js/screenshots…) n'ont pas d'équivalent DB → pas de divergence possible.

**#3 Clés + preflight** (`h4wk3y3.py`)
- `config/h4wk3y3.env` est déjà auto-chargé par `ArgusConfig` (copier le `.example`).
- `_preflight_report` : audit consolidé au démarrage — clés (6), outils externes (15),
  couverture. Ex : « 0/6 clés · 14/15 outils — manquant: testssl.sh ».

**#4 CVE activé + scheduler** (`core/scheduler.py`)

**#5 Signal/bruit findings** (`modules/m04_url_collector.py`)

**#6 OPSEC défauts** (`config/h4wk3y3.yaml`, `modules/m14_active.py`, `core/scope.py`, `modules/m13_nuclei.py`)
